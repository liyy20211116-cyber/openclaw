# OpenClaw 本地部署并绑定飞书机器人详细落地方案

## 1. 方案概览

本方案按“先可用、再稳态、后扩展”执行，分为 6 个阶段：

1. 环境与安全准备
2. OpenClaw 安装与模型认证
3. 飞书应用与机器人配置
4. 通道打通与联调验收
5. 稳态运行与运维保障
6. 迭代增强（可选）

---

## 2. 阶段一：环境与安全准备

### 2.1 目标

- 准备可运行环境（Windows + WSL2）。
- 完成凭据治理，避免敏感信息泄漏。

### 2.2 执行项

- 安装 WSL2（Ubuntu LTS）。
- 安装 Node.js `>=22`。
- 为项目创建本地运行目录（建议固定路径）。
- 在飞书开放平台轮换 `App Secret`（建议立即执行）。
- 创建本地 `.env`（仅本机可读，不入库）。

### 2.3 `.env` 模板（示例）

```bash
FEISHU_APP_ID=cli_xxx
FEISHU_APP_SECRET=xxxxxx
```

> 注：模型侧使用 OAuth 时，不需要 `OPENAI_API_KEY`；如果要做容灾，可另备 API Key 方案。

### 2.4 阶段验收

- WSL2 中可执行 `node -v` 且版本满足要求。
- 飞书 Secret 已轮换。
- `.env` 已创建且权限受控。

---

## 3. 阶段二：OpenClaw 安装与模型认证

### 3.1 目标

- 在本地安装并启动 OpenClaw。
- 完成 ChatGPT 账号 OAuth 认证（`openai-codex`）。

### 3.2 执行命令（WSL2）

```bash
npm install -g openclaw@latest
openclaw onboard --install-daemon
```

模型认证与默认模型设置：

```bash
openclaw models auth login --provider openai-codex
openclaw models set openai-codex/gpt-5.3-codex
openclaw models list
```

### 3.3 建议配置基线

- 主模型固定为 `openai-codex/gpt-5.3-codex`。
- 对群聊默认设置“需 @ 触发”。
- 设置白名单与基本限流策略。

### 3.4 阶段验收

- `openclaw models list` 可见已设置默认模型。
- `openclaw doctor` 无阻断性错误。
- Gateway 能正常启动。

---

## 4. 阶段三：飞书应用与机器人配置

### 4.1 目标

- 飞书应用具备机器人收发消息能力。
- OpenClaw 能使用飞书凭据建立通道。

### 4.2 飞书控制台配置清单

- 开启机器人能力。
- 开启事件订阅（按 OpenClaw 当前支持模式配置）。
- 开通必要权限（消息接收/发送为主，最小权限原则）。
- 将机器人添加到目标测试群。
- 配置可见范围（先测试用户/群，后扩展）。

### 4.3 OpenClaw 与飞书对接策略

- 飞书凭据由环境变量注入，不写死配置文件。
- 对群聊开启 mention gating。
- 启用 allowlist（用户/群）。

### 4.4 阶段验收

- 飞书机器人在线、可被拉入群。
- OpenClaw 启动日志中飞书通道初始化成功。

---

## 5. 阶段四：通道打通与联调验收

### 5.1 联调路径

1. 启动 Gateway（建议 verbose）
2. 飞书私聊发送“测试消息”
3. 飞书群聊不 @ 发送（应不触发）
4. 飞书群聊 @机器人 发送（应触发）
5. 连续多轮对话验证上下文

### 5.2 建议命令

```bash
openclaw gateway --port 18789 --verbose
```

### 5.3 验收指标

- 私聊 95%+ 请求在可接受时延内回复。
- 群聊触发逻辑符合预期（仅 @ 触发）。
- 失败请求可在日志中定位原因（鉴权/网络/模型/回发）。

---

## 6. 阶段五：稳态运行与运维保障

### 6.1 运行策略

- 使用 daemon 常驻。
- 开机自启。
- 保留最近 N 天日志。

### 6.2 可观测性

- 关键日志：入站事件、模型调用、出站回发、错误栈。
- 定期执行健康检查（如 `openclaw doctor`）。

### 6.3 安全策略

- 每 30~90 天轮换飞书 Secret。
- 定期复核飞书权限与可见范围。
- 对异常高频会话做限流和告警。

### 6.4 成本治理

- 仅允许白名单用户/群触发。
- 群聊必须 @ 才触发。
- 视情况配置回复长度、上下文窗口和重试策略。

---

## 7. 阶段六：迭代增强（可选）

- 增加多模型 fallback（当 OAuth 异常时切换 API Key provider）。
- 增加工具权限分级（私聊高权限、群聊低权限）。
- 增加知识库/FAQ 预处理降低 token 消耗。
- 增加 webhook 与业务系统联动。

---

## 8. 执行排期（建议）

- Day 1：环境准备 + OpenClaw 安装 + OAuth 认证
- Day 2：飞书配置 + 通道联调 + 验收
- Day 3：稳态治理（日志、告警、安全、成本）

---

## 9. 回滚预案

- 飞书通道异常：暂时关闭飞书事件订阅或移出机器人。
- 模型认证异常：切回备用 provider（API Key）保障连续性。
- 配置异常：恢复上一版配置快照并重启 Gateway。
